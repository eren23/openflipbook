## The two real issues

### 1. "Enter doesn't appear / re-tap needed" — ROOT-CAUSED on 2026-06-15
**→ full write-up: `docs/ENTER_RELIABILITY_2026-06-15.md`. Corrected conclusion below.**
- **It isn't really a 502, and it isn't a spurious-abort bug.** I instrumented all three abort
  sites and ran load repros (`scripts/record-demo/repro-502.ts`). Dominant cause = **view-critic
  *retry latency***: ~half of enters run two sequential ~40s pro edits (≈100s) because the critic
  rejects attempt 0 and retries. The backend completes fine; full completion is ~100s with a
  static banner, so it feels stuck — and a re-tap then aborts the in-flight retry. The
  `[ABORTDBG]` trace showed every abort was benign (stale-controller no-op) or a legit
  navigation.
- The **502** in the demo was a separate, rarer event — the background `extract` route under
  concurrent VLM load. It does not block the enter.
- **Fix is a latency/quality tradeoff** (touches the committed `enter_same_place` baseline), so I
  did NOT flip it overnight. Recommended: **`VIEW_LOOP_MAX_ATTEMPTS=1`**, eval-gated with
  `make eval-enter-drift`. Options + evidence in the dedicated doc — one word from you and I run
  the gate + ship-or-revert.

### 2. Geo overlay lags extraction
- Entity extraction localizes correctly (`extract.localized located:6/total:6`) but lands
  **~minutes** after the page renders (the city map's entities localized ~6 min later). So the
  `⊞ geo` overlay looks empty if toggled immediately. The data *does* persist.
- Nested entities use a **parent-local coordinate frame** (positions like -8400) vs the root's
  normalized 0–100 — the documented caveat, and why nested boxes don't draw cleanly on the
  current image. (`expected_layout` still doesn't steer pixels — also as documented.)
